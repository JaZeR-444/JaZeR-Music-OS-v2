01 — UX Verdict

Dashboard Score

UX Score
5.5

Structurally Confused
Verdict

The dashboard concept is pointed in the right direction — one question, filtered views, no reference material — but the section order is wrong, the information architecture confuses types of data with stages of action, and the key UX mechanisms are undefined at a Notion-build level.

The root problem: the current section order is organized around what kind of thing it is (Songs, Sessions, Ideas, Tasks, Releases) instead of what you do with it (Capture → Execute → Surface → Plan). When you open the dashboard asking "what do I work on right now?" you hit two passive information sections (Active Songs, Unattached Ideas) before you see a single actionable task. That's backwards.

"Content/Brand Tasks This Week" is the worst-named section in any Notion dashboard I can evaluate. It conflates two different task types (Content vs. Brand), adds a time qualifier ("This Week") that the underlying filter should handle, and uses a slash that signals internal confusion about the section's job. It will be renamed and rebuilt.

The dashboard is not clear enough for daily use in its current form. Nothing about the layout guides you from opening → deciding → acting. After corrections: 8.5/10, capped by Notion's inherent UX limitations (no custom animations, no conditional rendering, fixed block hierarchy).

View Type Decisions

Dashboard Section
Notion View
Why This View, Not the Others
Quick Capture
Template Button
Not a database view at all — a Notion Template Button block that creates a record and opens it immediately. Any alternative (link to DB, mention of a page) adds friction. The button is the only zero-friction mechanism Notion offers for record creation from a page.
Today's Focus
Table
Tasks are text records, not visual objects. Table view shows Task · Type · Priority in three narrow columns — fastest scanning format for action items. Board view would show task cards in columns which is too visual for a quick-decision surface. List view hides the Type/Priority which matters for context.
Active Songs
Gallery
Songs are creative projects that benefit from visual recognition — you know "Transformative Trance" the way you know a song, not a row in a table. Gallery shows 2 cards per row with Status, Priority, and Open Tasks as property chips. NOT board: board view on a dashboard requires horizontal scroll and shows pipeline stages you're not deciding between — you're just seeing what's active.
Unattached Ideas
Table
Ideas are text fragments, not visual objects. A table with two columns (Title + Asset Type) is the fastest way to scan 6–8 floating ideas. Gallery view would show blank cards. List view hides Asset Type. Table is correct.
Release Pipeline
Table
Releases have critical structured metadata — dates, countdowns, task counts, health status. Table shows 5 data columns at once. Gallery would hide the date/task data behind a click. Calendar view would be useful in Launch Hub for scheduling but not on the dashboard where you need urgency signals visible immediately.
Board view (anywhere)
Avoid on Dashboard
Board views use horizontal scroll, which is painful on a dashboard page and completely breaks on mobile. Boards belong inside hubs where they're the primary view and you're actively using them, not scanning them. The one exception: Song Lab uses a full-width board as its primary Songs view — that's the right place for it.
04 — Hub UX Review

Each Hub Evaluated

🎵 Song Lab
Strongest Hub · Refine Landing
Landing Experience

Cover image: yes. One dark, abstract waveform or moody music visual. Consistent with the workspace's premium feel. Full-width page layout for the board view to breathe.

Opening callout: One line. "Every song, every idea, every session." Not a paragraph of instructions — one line that orients without explaining.

Then: the two template buttons from the dashboard (Capture Idea + Log Session) replicated here. The hub should be as immediately actionable as the dashboard.

Primary View — Active Songs Board

Full-width Kanban board by Status. This is the right place for the board view — it's the primary working surface, not a scan. All 8 status columns visible. Song cards show: Title · Priority · Open Tasks count · Stale Warning.

The board in Song Lab is where you drag songs from Writing to Recording to Producing. The dashboard gallery is where you see which ones are active. Different jobs, different views.

Notion-Specific Elements
Full-width page toggle: ON
DB title: hidden on all linked views
Board: sort by Priority desc within each column
Secondary Views — Two-Column Layout Below Board

Left column: Writing Assets — two tabs on the same linked view: "Unattached" (Floating = true) and "Hook Bank" (Type = Hook). Switching tabs keeps you in the same column without navigating away.

Right column: Recent Sessions — a simple table showing the last 8 sessions (sort: Date desc). This is the correct home for session history — in Song Lab, not the dashboard.

The Book of JaZeR Sub-Page

Link to The Book of JaZeR as a prominent sub-page reference, not buried in the sidebar. A callout block with a direct link: "📖 Open The Book of JaZeR →". One block, one click.

What Must Not Appear Here

No Drive links. No plugin references. No release information. No brand assets. Song Lab is the writing room — every element should be about creating and developing songs. The moment a Drive link to a production guide appears here, it starts becoming a general reference page.

AI Agent Links

The Lyrics Refinement Agent and Song Development Agent pages should be linked in a collapsed toggle at the bottom of Song Lab labeled "Writing Tools →". They belong here but should not compete with the active work surface.

🎛️ Production Hub
Absorb → Not a Standalone Hub
The UX Argument Against This Hub

A hub page needs a reason to exist as a navigation destination — something that requires dedicated context you'd lose on the dashboard. Production Hub as described contains: a filtered Songs view (already on the dashboard as Active Songs) and a filtered Tasks view (already on the dashboard as Today's Focus). There is no third layer of context that justifies a dedicated page.

The test: if someone navigates to Production Hub after looking at the dashboard, what do they see that they couldn't see on the dashboard? If the answer is "nothing new," the hub is a redundant click.

What Production Hub Would Need to Exist

A legitimate Production Hub requires content that cannot live on the dashboard: song-level production checklists, A/B reference comparisons, LUFS tracking spreadsheets, plugin chain notes per song. If those exist, build the hub. If not, the hub is a folder masquerading as a workspace destination.

Recommended Resolution

Absorb production views into the dashboard and Song Lab. The "Production Queue" sections become dashboard Zone 2 (Today's Focus, filtered). The "Songs in production" becomes a sub-view in Song Lab's Active Songs board (filtered to Recording/Producing/Mixing statuses).

FL Studio Plugin CC link belongs in Zone 5 of the dashboard navigation footer and as a callout link inside each Song's page body. That's the complete integration surface — a linked sibling, not an embedded hub.

If You Still Want a Production Sub-Page

Create a sub-page inside Song Lab called "Production View" that shows Songs filtered to active production stages + linked production tasks. That's a sub-page of Song Lab, not a top-level hub. It doesn't get a sidebar slot.

🚀 Launch Hub
Conditional Value · Design for Dormancy
The Temporal Gap Problem

Between releases, Launch Hub is effectively empty. If you design it purely for active-release mode and don't address the dormant state, it will feel broken for most of the year. The hub needs two explicit design modes: Active Mode (release in progress) and Dormant Mode (between releases).

Dormant Mode Design

When no release is active (Releases DB Status ≠ In Prep, ≠ Submitted): the hub top shows a minimal callout — "No active release. When a release is within 60 days, this hub activates." Below it: one button to create a new Release record. Nothing else. Clean, intentional, honest.

Do not fill the dormant hub with decorative content, "coming soon" notes, or reference links. An empty hub that knows it's empty is better than a filled hub that looks abandoned.

Notion-Specific Tip
Hide DB views that show 0 results using a conditional callout above: "When a release appears here, this section activates." The DB view stays but the callout provides context for the empty state.
Active Mode Design (60 days before release)

Top section: Releases DB — single active release card with all fields visible: Status, Release Date, Days Until, Health badge, Song Count, Open Tasks. One record in focus.

Middle section: Release prep tasks (Tasks DB filtered to Type = Release Prep + linked to this Release). Table view, Status column prominent. Checkmark-driven.

Lower section: Content tasks (Tasks DB filtered to Type = Content). Same table format. Then: link to Content Calendar Generator page. Then: IG Release Caption examples (linked sub-pages, not embedded).

What Doesn't Belong Here

No Drive links to the 90-Day Launch report or Waterfall Release Ecosystem embedded inline. Those are reference documents — link to them from a collapsed toggle labeled "Release Strategy Docs →" at the bottom of the hub. They're not part of the daily workflow surface.

🏷️ Brand Hub
Demote to Reference Page · Not a Hub
Why It Fails the Hub Test

A hub requires: a primary database or workflow surface, recurring use (daily or weekly), and content that requires navigation context (you need to be "in" the hub to do the work). Brand Hub fails all three. About JaZeR is a reference document. Persona reports are Drive PDFs. The website is an external link. None of these require a dedicated navigation destination.

The consequence of keeping it as a hub: it occupies a sidebar slot that builds a false expectation of daily utility. You click it twice, find reference docs, and stop going there.

Promotion Criteria

Brand Hub earns hub status when: active website development tasks are tracked here (Brand tasks in Tasks DB, volume > 10/month), a content voice guide is actively referenced during song writing sessions, or a brand refresh project is in progress with open decisions.

Recommended: Brand Reference Page

Create one page called "Brand Reference" — pinned in the sidebar, not a top-level hub. Structure it as three sections:

Identity: About JaZeR (linked), one-line persona/archetype summary extracted from the Drive reports, artist voice note (2–3 sentences about the creative identity — written once, rarely updated).

Website: JaZeR Brand Website link, live site URL, website status note.

Drive Research: Linked list of Listener Persona, Archetype, Ideal Fan, and 90-Day Launch report. No embedded content — just clean linked references.

Notion Placement
Location: Sidebar "Favorites" or pinned link in dashboard Zone 5 footer
Page width: Normal (not full — it should feel compact)
Cover: No cover — pure reference pages don't need covers
📦 Vault
Right Concept · Design for Friction
Design Principle — Make It Deliberately Less Inviting

The Vault should feel slightly more effort to enter than the active hubs. This is intentional — you want a small amount of friction that makes you pause and ask "do I actually need to go here, or am I avoiding active work?"

Implementation: no cover image. No decorative elements. Plain page width (not full). Muted icon (📦, not something exciting). A pinned callout at the top with the Phase 3I Archive Protocol governs this page — linked, not embedded.

Structural Design

Three toggle sections — no open content. Every legacy item lives inside a collapsed toggle. You only see the category label until you open it. This prevents the Vault from looking like active content.

Toggle 1: Legacy Hubs — Funky Futuristic CC, Music Hub v1.0. Each with: name · what it was · date archived. No embedded views.

Toggle 2: Backup Tree — JaZeR Musician Page, 10_Music_Creative_Projects, JaZeR Life OS Architecture. Same format.

Toggle 3: Planning Docs — Old architecture plans, structure snapshots, Mermaid Planning Pack v1. Same format.

The Gate — Archive Protocol

The top of the Vault page shows one callout block: "Vault governance: Phase 3I Archive Protocol Plan →" with a link to the existing protocol page. Everything that enters the Vault must pass through this protocol. The callout is the visual gate — you see it every time you enter the Vault.

No Embedded Databases in the Vault

Do not embed linked database views from legacy systems inside the Vault. Page links only. If you embed a view of the old Songs DB from Funky Futuristic, it makes the Vault feel like an active system with live data. The Vault is a list of links, not a workspace.

Access Path

The Vault is in the dashboard Zone 5 navigation footer as the last link. It is not in the sidebar favorites. Finding it requires going through the dashboard — the extra navigation step is the friction mechanism.

Notion Placement
Page width: Normal
Cover: None
Lock icon in title: optional but effective signal
All content inside collapsed toggles: required
05 — Visual Hierarchy Recommendations

Notion-Specific Design Rules

⚠️

The Notion Museum Syndrome warning: This is when a workspace becomes more beautiful to look at than useful to work in. The signs: gallery views of everything, a different icon for every block, colored dividers between every section, property badges everywhere, cover images on every sub-page. The treatment: reserve all decoration for hub pages. The dashboard should be spare. The work should be visible, not the system.

Page Title
One per page. No subtitle. No status note.
Dashboard: "JaZeR Music OS" — nothing else. Hub pages: "Song Lab" · "Launch Hub" · "Brand Reference." Never put the date, version number, or emoji in the page title. The icon carries the identity. The title states the purpose.
✕ Avoid: "🎵 Song Lab v2 · Updated May 2026"
Section Headers
All-caps mono text OR Notion H3. Never H1/H2 within a page body.
On the dashboard, section labels should be visually subordinate to the content they introduce. Use a small all-caps mono text block or a lightweight Notion H3 (gray, not the default black) — not bold H2 headers that compete with the database views for visual weight. The content is the star.
✕ Avoid: Using H1/H2 inside dashboard body — reserve H1/H2 for document-style hub pages only
Callout Blocks
One per section maximum. Reserved for actions and alerts only.
Callout blocks are visually prominent — they draw the eye. If every section has a callout, none of them work. Use callouts ONLY for: (1) the Quick Capture buttons in Zone 1, (2) the "No active release" dormant state in Launch Hub, (3) the Vault governance gate. Three callouts across the entire workspace. No more.
✕ Avoid: Informational callouts that describe what the section does — the section label and content should be self-explanatory
Dividers
Horizontal rule only. No colored text dividers.
Use Notion's native divider (---) between major dashboard zones. Not colored, not emoji-bordered, not heading-styled. A simple gray line. The space itself creates separation; the divider just makes it explicit. Inside hub pages, use spacing (empty lines between sections) rather than dividers — it feels lighter.
✕ Avoid: "═══ 🎵 SONGS ═══" or colored callout blocks used as dividers
Database View Titles
Always hidden on dashboard. Visible only inside hub pages.
When you embed a linked database view, Notion shows the database name above it by default. Hide this — it creates visual clutter and states what your section label already states. The section header IS the view title. Inside hub pages where multiple views of the same DB are shown, keep the view name visible to distinguish between views.
✕ Avoid: "Songs — Active Songs Board View" header above an embedded view that's already under a "ACTIVE SONGS" section label
Icons & Covers
One emoji per hub page. No icons inside page bodies.
Page icon (the emoji that appears in the sidebar and at the top of the page): one per hub, consistent set. Dashboard: 🎯 Song Lab: 🎵 Launch Hub: 🚀 Brand Reference: 🏷️ Vault: 📦. Cover images: Dashboard (yes — abstract dark image). Song Lab (yes — same style). All other pages: no cover. The cover's job is to signal "this is an important destination," not "this is pretty."
✕ Avoid: Different emoji icon per section within a page body · Cover images on reference pages and sub-pages
Synced Blocks
One legitimate use case: status snapshot across hubs.
A synced block is a block that appears identically on multiple pages. The one place this is genuinely useful: a "Current Focus" one-liner that appears on both the dashboard and Song Lab. Update it once (e.g., "Focus: Cosmic Beam release prep · Mixing Transformative Trance") and it's current everywhere. Do not over-use synced blocks — they create hidden dependencies that become confusing.
✕ Avoid: Syncing entire sections between pages — if two sections need the same content, the content belongs in one place and should be linked, not synced
Database Properties on Views
Show only decision-relevant properties. 2–4 columns max on dashboard views.
When configuring a linked database view, every property you show is a column the eye has to process. On the dashboard: Songs gallery shows Status + Priority + Open Tasks (3 properties). Tasks table shows Task + Type + Priority (3 properties). Ideas table shows Title + Asset Type (2 properties). Release table shows Title + Status + Date + Days Until + Health (5 — the maximum). Any more than 5 visible properties on a dashboard view is cognitive overload.
✕ Avoid: Showing BPM, Key, LUFS, and ISRC on the dashboard Songs view — those belong on the Song record page, not the dashboard card
06 — Friction Audit

What Slows You Down

🔴
CRITICAL — Quick Capture is undefined at the mechanism level
The dashboard lists "Quick Capture" as a section but the actual Notion mechanism isn't specified. If capture requires: navigating to Writing Assets DB → clicking New → selecting a template → filling in properties → returning to dashboard, that's 6–8 steps. In a creative moment, anything more than 2 steps loses the idea. The mechanism must be a Notion Template Button block directly on the dashboard page that creates a record AND opens it with one click.
Fix: Use Notion's native "Button" block (⌘ + / → Button) or a Template Button linked to Writing Assets with pre-filled Asset Type = Idea, Status = Raw, Date Created = auto. Opens immediately on click. Total friction: 1 click + type the idea.
🔴
CRITICAL — Session logging has no zero-friction path
The current dashboard has no "Log a Session" button. Logging a session requires navigating to Sessions DB → creating a new record → selecting template → filling in 3+ fields. After a creative session when momentum is high, this path gets skipped. The dashboard must have a session log button alongside the capture button — same zone, same level of access.
Fix: Second template button in Zone 1 — "Log a Session" — creates a Session record with Session Type (select prompt) and Date = today pre-filled, Song relation prompted on open. Total friction: 1 click + select type + select song.
🔴
CRITICAL — "Weekly Review Anchor" is two text rows going nowhere
The current design shows: "What got done this week?" and "What's the top priority next?" as text rows on the dashboard. These are questions, not systems. They don't link to anything. They don't generate a page. They don't pull from data. After the first week, you stop reading them because they never change and nothing happens when you read them.
Fix: Replace with a Notion Button block labeled "Start Weekly Review →" that creates a new page from a Weekly Review template. The template auto-populates the date and links to filtered views of: songs advanced this week, sessions logged, tasks completed, open focus tasks, and a "Next priority" prompt. One click creates a real review session, not a reminder to have one.
🔴
HIGH — Section order doesn't match the decision sequence
Current order: Capture → Active Songs → Unattached Ideas → Production Queue → Recent Sessions → Release Pipeline → Content Tasks → Weekly Review. The decision sequence should be: Capture (if needed) → Execute (what am I doing now) → Context (which songs, what's floating) → Plan (what's coming). Putting Active Songs before Today's Focus means you process an entire song gallery before you see the specific tasks you're doing today. That's backwards — you decide what to DO before you contextualize WHERE.
Fix: Reorder to the five-zone system: Capture → Today's Focus → Active Songs + Unattached Ideas (side by side) → Release Pipeline → Hub Navigation.
🟡
MEDIUM — No Focus task path from the dashboard
Today's Focus shows tasks where Focus = true. But how does a task get Focus = true? There's no mechanism for this on the dashboard. The workflow requires: go to Tasks DB or Song page → find a task → open it → toggle Focus = true → return to dashboard. That's 4 steps to surface one task. If the friction of marking Focus is higher than the value of the Focus view, it won't be used.
Fix: Inside each Song page, add a Tasks linked view with a "Mark as Focus" action. The best Notion solution is a button inside each Song record that creates a new task with Focus = true pre-checked. Alternatively, configure the Task DB to show Focus as a checkbox column in table views — one click to check it. Make it a 1-click action from wherever you are when you decide what to work on.
🟡
MEDIUM — Content/Brand Tasks section name signals confusion
"Content / Brand Tasks This Week" is a section name that reveals the system hasn't decided what it is. The slash says "I couldn't choose between content and brand." "This Week" says the filter is doing the label's job. A section that can't describe its own purpose in 2 words is a section that needs to be rethought.
Fix: This section doesn't exist in the new layout — all task types surface through the Today's Focus view (Focus = true across all Task Types). Content tasks appear there when they're marked as Focus. Separate task-type views belong in Launch Hub, not the dashboard.
🟡
MEDIUM — No empty state design for Release Pipeline
Between release cycles, the Release Pipeline section shows an empty table with a "No pages found" state (Notion default). This makes the section look broken rather than dormant. A broken-looking empty state is worse than no section at all — it erodes trust in the system.
Fix: Place a callout block above the Release Pipeline table with conditional intent: "Release Pipeline activates when a release is within 60 days." When it's empty, users understand why. When it has data, the callout is invisible context. Between releases, consider hiding this zone entirely behind a toggle.
🔵
LOW — Hub navigation footer has no visual differentiation from body content
If the hub navigation links at the bottom of the dashboard look like regular text or regular Notion links, they blend into the page body and don't feel like navigation controls. Users may scroll past them without registering that they're navigation.
Fix: Style each hub link as a small "button-style" block — a callout with a background color (subtle, consistent), or use a 5-column layout with each column holding one hub link styled as a mini-card. Make them feel visually distinct from the database views above them.
07 — Mobile Usability Audit

What Breaks on Mobile

📱

Notion mobile critical behavior: All column layouts collapse into a single vertical stack, left column then right column. There is no responsive layout control — you cannot tell Notion "show column A before column B on mobile" beyond the column order you set on desktop. Mobile also does not support hover states, and database views are smaller and require more precise tapping. Build the desktop layout with mobile stack order in mind.

✓ Works Well on Mobile
Template Buttons (Zone 1) Full-width tap targets. One of the best Notion mobile interactions — large, clear, immediate. Quick Capture and Log Session buttons work perfectly on mobile.
Today's Focus Table (Zone 2) Table with 2–3 narrow columns reads well on mobile. Task title is most important and appears first. Tapping opens the full task record.
Unattached Ideas Table (Zone 3R) Simple 2-column table (Title + Asset Type) renders well on mobile. Fast-scan format that doesn't require horizontal scrolling.
Release Pipeline Table (Zone 4) 5 columns is near the mobile limit. Renders horizontally scrollable which is acceptable for a planning view you check less frequently than action views.
Hub Navigation (Zone 5) Collapses to a vertical stack of 5 button-style blocks. Each is a large tap target. Works perfectly on mobile.
✕ Breaks or Degrades on Mobile
Two-column layout (Zone 3) Desktop: Active Songs (left) and Unattached Ideas (right) side by side. Mobile: Active Songs stacks on top, Unattached Ideas below. This is acceptable IF column order is correct — Songs before Ideas is the right stack order for mobile priority. Verify this in build.
Gallery view — Active Songs A 2-per-row gallery on desktop becomes 1-per-row on mobile, making it scrollable. 4 song cards become 4 separate rows — still readable but requires scrolling. If this becomes a problem, use a table view as the mobile fallback.
Board views (anywhere) Kanban boards require horizontal scrolling on mobile — extremely painful. Never use Board view on the dashboard. Keep Board view inside Song Lab where it's the intentional primary working surface, even if that's also painful on mobile.
Database view property columns Views with 4+ visible property columns will require horizontal scroll on mobile. Keep dashboard views to 3 columns max for mobile comfort. The Release Pipeline at 5 columns is the exception — it's used for planning, not daily execution.
△ Design Decisions for Mobile
Zone 2 is the most important mobile zone On mobile, Today's Focus is the primary daily interaction. It must render immediately above the fold after Zone 1 (Capture buttons). On a phone screen, Zone 1 + Zone 2 should be visible without scrolling — or Zone 1 at minimum. Design Zone 1 to be compact (2 buttons side by side, not stacked).
Column order determines mobile stack order Zone 3 two-column layout: put Active Songs in the LEFT column on desktop so it stacks ABOVE Unattached Ideas on mobile. Active Songs is more important for daily decisions — it should come first in the mobile stack.
Gallery card height on mobile Property-only gallery cards (no covers) render shorter on mobile than on desktop. 4 songs = 4 compact cards ≈ one screen on most phones. That's acceptable. Gallery with cover images would be too tall — confirms the "no covers on dashboard gallery" rule.
Create the workspace on desktop, use it on mobile Notion databases are built and configured on desktop. Use mobile primarily for capture (Quick Capture button), quick task status updates (Done checkbox in Today's Focus), and checking the release pipeline. Deep database editing, view configuration, and hub work happens on desktop.
△ Mobile-First Quick Capture
The most important mobile flow Idea → tap "Capture Idea" button → new Writing Asset opens → type the idea title → done. This must work with zero friction on mobile. Test this flow specifically on iPhone/Android before launch.
Session log on mobile The "Log a Session" button must default Date to today and prompt only Session Type + Song. Do not require more than 3 taps after button press before the record is saved. Everything else is optional and can be added later from desktop.
Notion sidebar on mobile Notion mobile shows the sidebar as a slide-out panel. Hub navigation in Zone 5 is less relevant on mobile because the sidebar already provides hub access. Zone 5 is primarily a desktop convenience.
Bookmark the dashboard Add the JaZeR Music OS dashboard to your phone's home screen or pin it in Notion's mobile "Favorites." The fastest mobile workflow is: unlock phone → tap dashboard → tap Capture or check Focus. Any additional navigation before that is friction.

Dashboard Build Rules — Non-Negotiable

RuleWhy
5 zones maximumAny zone above 5 competes with zones 1–2 for attention. The dashboard answers one question — it doesn't need more than 5 sections to do that.
No database title visible on any linked viewThe zone label is the title. Showing both creates redundancy that signals "this wasn't designed, it was assembled."
No board views on the dashboardHorizontal scroll on dashboard = friction. Board views belong in hubs where they're the primary working surface.
Zone 1 must be interactive, not informationalThe first thing you interact with sets the tone of the session. A read-only status summary is passive. A template button is active. Start with action.
Zone 2 must show only Focus = true tasksIf Zone 2 shows anything other than explicitly flagged focus tasks, it becomes a full task list. That's a different page. The dashboard shows focus, not everything.
Empty states must be intentional, not defaultNotion's default "No pages found" empty state is jarring. Every DB view on the dashboard should have custom empty state text that explains what the empty state means (success or dormancy — not an error).
Cover image: yes. Decorative body elements: no.The cover image gives the dashboard a premium feel from the moment it loads. Decorative elements inside the body (colored dividers, icon-heavy headers) compete with the data. One in, one out.
Column order = mobile stack orderLeft column stacks before right column on mobile. Active Songs must be LEFT so it stacks above Unattached Ideas on mobile. Verify this in the Notion mobile app immediately after build.

The test for a finished dashboard: Open Notion cold — no context, no recent history in mind. From the moment the dashboard loads, you should be able to answer "what do I work on right now?" within 10 seconds without scrolling below Zone 3. If you can't, the dashboard isn't done. The five-zone layout above passes this test. Zone 2 (Today's Focus) shows your tasks. Zone 3L (Active Songs) shows which project. Zone 3R (Unattached Ideas) shows what needs attention.
