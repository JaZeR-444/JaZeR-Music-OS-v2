Floating Writing Assets DB · Depends on: Song (relation) · Status
Boolean that returns true when a writing asset is unattached (no Song) and not Archived. Powers the Unattached Ideas default view without requiring the same complex filter to be recreated on every view. If Floating = true, the asset needs attention — it's a live fragment with no home.
and( empty(prop("Song")), prop("Status") != "Archived" )
Dependencies: Song relation + Status property. Both must exist before building this formula. Build order: Song property first → Status property → then add Floating formula. The Unattached Ideas view filter uses: Floating = true. This is valid Notion formula syntax in both the current and legacy formula engines.

Asset Type Definitions — All 7 Confirmed Types

TypeWhat It IsTypical Evolution
Idea A raw fragment — a word, title, vibe note, or concept sentence. Unnamed, undeveloped. Use Status = Developing when it grows beyond this. → Hook, Verse, or attached to a Song
Hook A melodic or lyrical phrase with strong potential — the part people remember. Usually 1–4 bars. Can precede a Song record or attach to one in progress. → Chorus; attached to a Song at Writing stage
Verse A completed or near-complete verse draft, typically 8–16 bars. Attached to a specific song once committed. → Part of a Lyric Sheet
Chorus A completed or near-complete chorus. Often the last section to be finalized; hooks often become choruses. → Part of a Lyric Sheet
Bridge A transitional section draft — contrast material, breakdown, spoken word, or emotional turn. → Part of a Lyric Sheet
Lyric Sheet The complete assembled lyric document for a finished song. Terminal type — created at Writing or Recording stage, attached to exactly one Song. → Terminal; attached to Released song
Freestyle Unstructured, raw session output — typed or transcribed during a session. High volume, low pressure. May never evolve; may become the best hook in the catalog. → Verse, Hook, or Idea; or Archived

Open Questions 3 Items — Answer Before Approving Phase 2A 3 items
OQ-1 Do formula properties count against the 12-property cap?
Songs DB has 12 data properties (rows 1–12) plus 1 formula (Status Progress %) = 13 total Notion property columns. The formula reads existing data and outputs a computed number — it adds zero data-entry burden. However, it does count as a visible column in Notion's property panel. If the rule is "12 total including formulas," one data property must be cut to make room for the formula.
Recommendation: Formulas count separately from data properties. The cap of 12 applies to data-entry properties (rows 1–12). Formulas are display utilities, not data fields. If this is approved, Songs DB launches with 12 data properties + 1 formula = 13 total columns. If this is not approved, the lowest-priority data property to cut first would be FL Studio Path (it can live in the Song page body instead), bringing data properties to 11 + 1 formula = 12 total.
OQ-2 Writing Assets Status: 3 options (Raw · Developing · Archived) or 5 options from the DB Schema?
The DB Schema source document specified 5 Status options: Raw → Developing → Attached → Used → Archived. Phase 2A proposes 3 options by cutting "Attached" (redundant with the Song relation field) and "Used" (inferred from the linked Song's Released status). The 5-option version gives more granularity — you can distinguish "attached but not yet used in recording" from "recorded and used." The 3-option version is simpler and relies on the Song relation to carry attachment state.
Recommendation: 3 options (Raw · Developing · Archived). "Attached" is covered by whether the Song relation is filled. "Used" in a released song is covered by the linked Song's Status = Released. Adding redundant status options creates maintenance overhead — you'd need to remember to update Status when you fill the relation. The relation is the ground truth; Status tracks development maturity only.
OQ-3 Songs DB Key property: 12-option root-note Select, or free Text?
The current spec uses a 12-option Select (C, C#, D, Eb, E, F, F#, G, Ab, A, Bb, B). Major/minor is noted in the Song page body template. This is filterable ("show me all songs in F#") but loses major/minor in the database column view. The alternative is a Text property ("F# minor") — fully expressive but not cleanly filterable or sortable. A third option is two properties: Root Note (12-option Select) + Mode (2-option Select: Major/Minor) — fully expressive and filterable, but uses 2 property slots instead of 1 (bringing data properties to 13).
Recommendation: Single 12-option root-note Select. Major/minor in the Song page body is sufficient for a solo artist's catalog. Filtering by "show me F# songs" is more useful than filtering by "show me minor key songs" at this catalog size. Two-property option deferred to Day 30+ if needed. Confirm the single Select, or approve the two-property approach (noting it adds 1 data property).
