02 Conflicting Instructions

Must Be Resolved Before Build

These are genuine contradictions between documents — not differences in emphasis, but different instructions that would produce different builds. Each has a recommended resolution. Your approval of the resolution closes the conflict.

Build-Blocker C-01 · Songs DB Status First Step: "Idea" in Implementation Plan vs. "Concept" Everywhere Else
Implementation Plan — Phase 2 Prompt (P2)

The status flow is stated as: Idea → Writing → Recording → Producing → Mixing → Review → Release Prep → Released. "Idea" appears as the first status option without the rename.

Architecture Review · DB Schema Audit · Workflow Audit

All three review documents agree: Song Status "Idea" must be renamed to "Concept" to eliminate the naming collision with Writing Assets Type = "Idea." These are different objects.

Resolution
The correct status flow is: Concept → Writing → Recording → Producing → Mixing → Review → Release Prep → Released. The Implementation Plan Phase 2 prompt (P2) must be updated before use. Claude must use "Concept" as the first Songs DB status in all Phase 2 work. The Phase 5 prompt that says "Status is not Idea" must also be updated to "Status is not Concept."
Build-Blocker C-02 · "Unattached Ideas" Filter: Status = Active vs. Status ≠ Archived
Implementation Plan — Initialization Prompt

Filter defined as: linked Song = empty AND Status = Active. This is restrictive — only shows records where Status is exactly "Active."

Architecture Review — DB Section 5

Filter defined as: Song relation is empty, Status ≠ Archived. This is permissive — shows everything except explicitly archived records.

Resolution — Requires Your Decision
These produce different result sets. "Status = Active" hides records in "Developing," "Raw," "In Progress," or any other non-Active status — which means a hook you're actively working on might not appear in this view. "Status ≠ Archived" shows all in-progress work regardless of status label, which is more operationally useful. Recommended: use Status ≠ Archived. It shows the full creative inbox except things you've explicitly put away. Confirm this before Phase 2.
Fix Required C-03 · Sessions DB Property Cap: "3 required fields" vs. "cap at 5"
Implementation Plan — Phase 2 Prompt (P3)

States: "Hard cap: 5 properties maximum." Framed as a property count ceiling for the entire database schema.

Architecture Review · DB Schema Audit · Workflow Audit

States: "3 required fields only" with all others optional below the fold. Framed as a required-field floor, not a total cap.

Resolution
These are compatible but need unifying language. The correct spec is: Sessions DB has exactly 3 required database-column properties (Date, Type, Song). Additional optional properties — up to a total of 5 columns — may exist but should be hidden by default. The Implementation Plan's "cap at 5" should be reread as a total column cap. The Architecture Review's "3 required fields" describes the template behavior. Both can be true simultaneously. The Phase 2 Sessions prompt must clarify: 3 required, up to 5 total, nothing else without permission.

Fix Required C-07 · Tasks View Primary Filter: One Combined View vs. Two Separate Views
Implementation Plan — Phase 5 Chain Prompt (P6)

States: default Tasks view = "This Week" filtered to Focus = true OR Due Date within 7 days. One view with OR logic combining two different priorities.

Architecture Review · Workflow Audit

Specifies two separate dashboard views: (1) "Focus Tasks" filtered on Focus = true, Status ≠ Done — for execution. (2) "Due This Week" filtered on Due Date ≤ 7 days, Status ≠ Done — for planning. Different purposes, different views.

Resolution
Two views is correct. A single OR-logic view conflates execution priority with calendar priority — they are not the same thing. A task due in 3 days might not be what you work on today; a Focus-checked task might not have a due date. The Phase 5 chain prompt must be updated to specify two separate Tasks views on the dashboard: "Focus Now" (Focus = true) and "Due This Week" (Due Date ≤ 7 days). The Tasks DB default view when opened directly from the sidebar should be "Focus Now."

03 Terminology Corrections

Canonical Vocabulary for the Build

Five terms are used inconsistently across the documents. Any prompt sent to Claude must use the terms in the "Use This" column. Mixed terminology causes Claude to produce ambiguous specs and mis-classify pages.

Term Was Used As Problem Canonical Definition
Hub All 5 sidebar pages, including Brand Hub and Production Hub Calls pages "hubs" even when they have no active workflows, creating a false equivalence between Song Lab (daily use) and Brand Hub (quarterly reference). A Hub is a first-level sidebar page that: (1) contains at least one active database view, (2) serves a daily or weekly workflow, and (3) would be opened at least once per week during active use. Currently: Song Lab, Launch Hub, Vault.
Reference Page Sometimes "Reference Folder" (Implementation Plan) or "Reference Page" (Architecture Review). Both refer to Brand Hub's demoted state. Two terms for the same thing across the same document set causes confusion about whether Brand Hub is a folder (like a sidebar group) or a page (like a Notion page). A Reference Page is a single Notion page in the sidebar that contains only: static reference links, external URLs, Drive file links, and embedded documents. No database views. No recurring workflow. Brand Hub becomes a Reference Page — one Notion page, not a folder.
Dashboard Section Sometimes used for Production Hub's demoted state (correct), but also used loosely for any section heading on any page. Without a precise definition, "dashboard section" blurs into "hub page section" and Claude may not know whether it's specifying the Master Dashboard layout or a hub page layout. A Dashboard Section is a named block on the Master Dashboard page that contains exactly one filtered database view, a section heading, and optionally a description. Production Hub becomes two Dashboard Sections: "Songs in Production" and "Production Tasks." These live on the Master Dashboard only — not on any hub page.
Hub Page Section Rarely used — hub page layout sections are implicitly described but not given a distinct name from Dashboard Sections. Without this term, specifications for hub page layout and specifications for dashboard layout use the same vocabulary, making it possible for Claude to mix them up in Phase 3. A Hub Page Section is a named block within a hub page (Song Lab, Launch Hub, Vault) that contains a database view filtered to that hub's scope. Hub Page Sections are different from Dashboard Sections — they are themed to their hub context, not optimized for the "what do I work on right now?" question.
Unattached Ideas Used as both a view name and a filter description interchangeably, with two different filter definitions (see Conflict #2). The term "Unattached Ideas" conflates the view name with a filtering concept. The filter conditions have been defined two different ways in two documents. Unattached Ideas is a view name only. The canonical filter conditions (per Conflict #2 resolution) are: Song relation is empty AND Status ≠ Archived, sorted by Date Created descending. This is the Writing Assets DB default view. It is not called "All Ideas" or "Inbox" — it is always called "Unattached Ideas."
