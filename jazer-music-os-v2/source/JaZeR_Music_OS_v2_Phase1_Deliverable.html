Part A

Confirmed Decisions

Every item below is stated as a locked fact. These decisions are not open for reconsideration in Phase 2 or later phases unless a written architectural revision is issued.

Active Hubs (3)
—
🎵 Song Lab The primary daily creative workspace where songs are written, developed, and tracked through the full pipeline from Concept to Release Prep.
—
🚀 Launch Hub The release pipeline surface — activated when a release is 60 days out — housing release records, distribution prep tasks, and content scheduling.
—
📦 Vault The hard archive boundary for legacy content, inactive resources, and completed projects, governed by the Phase 3I Archive Protocol Plan.
Production Hub Disposition
Production Hub is not a sidebar hub and has no sidebar page. Its content becomes two named Dashboard Sections on the Master Dashboard:
— Songs in Production — filtered Songs DB view (Status = Recording, Producing, or Mixing)
— Production Tasks — filtered Tasks DB view (Type = Production, Status ≠ Done)
Rationale: two filtered views and one external link do not constitute a hub. Absorbing them into the dashboard eliminates a navigation layer that adds no organizational value.
Brand Hub Disposition
Brand Hub is not a sidebar hub and has no sidebar page. It becomes one Reference Page pinned in the sidebar:
— Brand Reference — single Notion page containing: About JaZeR link, Drive persona/archetype/ideal fan report links, JaZeR Brand Website URL
Promotion criteria: Brand Reference is promoted to Hub status only when active brand workflow volume develops (e.g., active website development tasks, content voice guide in active use, brand refresh project in progress).
Vault Gate Criteria
A resource moves to the Vault when all three conditions are met:
1 No active workflow in v2 — the resource is not used in any current database, hub, or recurring workflow.
2 Overlaps with a confirmed v2 database or hub — its function is fully covered by the new system.
3 Not opened in 30 days — or it is a confirmed legacy/backup page, in which case the 30-day condition is waived.
Governing document: Phase 3I Archive Protocol Plan. Lives linked as the first visible element at the top of the Vault page — before any other content. This document is the decision authority for all archiving. If in doubt, consult Phase 3I before moving anything.
Master Dashboard Rule
The Master Dashboard contains only filtered views of the five core databases — no records, documents, notes, or content of any kind lives on the dashboard itself.
Corollary: anything that is not a filtered database view does not belong on the dashboard. The single exception is the Quick Capture template button, which is a Notion button (not content) that creates a Writing Assets record without navigating away.
Scope Exclusions
The following are excluded from JaZeR Music OS v2. They are not deferred — they are out of scope unless a new explicit written request is made:
✗ Finance / income / expense tracking
✗ Collaborator CRM
✗ Touring / live performance database
✗ Sample library / sounds database
✗ FL Studio Plugin Command Center — remains an external linked sibling; never absorbed into Music OS v2
✗ Any 6th database — requires explicit written approval with a stated workflow justification
Part B

Canonical Vocabulary

These five definitions apply to this workspace only. Every phase prompt, deliverable, and spec must use these terms exactly. If a term is used differently in any document, the definition below is authoritative.

Hub
A first-level sidebar page that: (1) contains at least one active database view filtered to its scope, (2) serves a daily or weekly workflow that requires its own navigation destination, and (3) is expected to be opened at least once per week during active use. A page with only reference links, external URLs, or archived content is not a Hub.

Confirmed Hubs: Song Lab Launch Hub Vault
Reference Page
A single Notion page in the sidebar containing only static reference links, external URLs, Drive file links, and optionally read-only embedded documents. It has no database views, no recurring operational workflow, and is not the primary surface for any active task. It is navigated to when you need to look something up — not when you need to do work.

Confirmed Reference Page: 🏷️ Brand Reference
Dashboard Section
A named block on the Master Dashboard page containing exactly one filtered database view and one section heading. Dashboard Sections exist solely to answer "What do I work on right now?" Every Dashboard Section must pass this test before it is built. They appear on the Master Dashboard only — not on Hub pages.

Confirmed Dashboard Sections replacing Production Hub: Songs in Production Production Tasks
Hub Page Section
A named block within a Hub page (Song Lab, Launch Hub, or Vault) containing a database view filtered to that Hub's scope and purpose. Hub Page Sections are designed around that Hub's specific workflow context — they are not governed by the "What do I work on right now?" constraint and may show historical, planning, or reference-adjacent content appropriate to their Hub.

Hub Page Sections and Dashboard Sections are distinct. A view appearing on the dashboard and the same view appearing in Song Lab are different sections with potentially different filters, sorts, and visible properties.
Unattached Ideas
The name of the default view for Writing Assets DB. It functions as the creative inbox — the first thing you see when you open Writing Assets. It is never called "All Records," "Inbox," or "Ideas."
Filter: Song relation = empty AND Status ≠ Archived
Sort: Date Created — Descending
Position: View #1 in Writing Assets DB (opens by default)

↳ 🎵 Songs DB Database

Level 2 — Inside Launch Hub

↳ 📁 Archive Batch Sub-Pages — Funky Futuristic CC, Freestyle Hub V1, backup tree, Music Hub v1.0
Tasks DB at Level 1 — rationale: Tasks spans Production, Release Prep, Content, Brand/Website, Admin, and Review. No single hub owns more than a portion of it. Nesting Tasks inside Song Lab or Launch Hub would misrepresent its scope. Placing it at Level 1 as a standalone cross-hub database is clean, honest, and avoids the perception that tasks are owned by any one workflow context. Hub pages and the Master Dashboard access it exclusively through embedded filtered views.

Open Questions 4 Items — Require Your Decision Before Phase 2 Begins 4 items
OQ-1 Sessions DB — confirm the Session Type options
Two different option sets exist across the source documents and cannot both be correct. The Implementation Plan (P3) lists: Songwriting / Production / Mixing / Other. The Database Schema Audit lists: Writing · Freestyle · Recording · Production · Mix Review. These are meaningfully different — the Schema Audit version distinguishes "Recording" (vocal/instrument capture) from "Production" (beat-making, arrangement) and has "Mix Review" as a distinct session type rather than just "Mixing." The Implementation Plan version is simpler but less specific.
Recommendation Use the Database Schema Audit option set: Writing · Freestyle · Recording · Production · Mix Review. It maps more precisely to the actual sessions a solo recording artist has — "Writing" and "Freestyle" are meaningfully different session types, and "Mix Review" as a named type makes it easy to track how many mix passes a song took. Confirm or adjust before Phase 2.
OQ-2 Tasks DB — confirm the Task Type options
The confirmed Type list from the Implementation Plan is: Production · Release Prep · Content · Brand / Website · Admin · Review. This has never been explicitly approved by you. Phase 2 schemas the Tasks DB and the Type property's option list must be locked before that work begins. If any Type should be renamed, consolidated, or removed, now is the correct time to do it — not after the schema is built.
Recommendation Confirm the list as-is: Production · Release Prep · Content · Brand / Website · Admin · Review — 6 Types. "Review" covers weekly reviews, listening sessions, and feedback passes. If "Admin" feels too vague, consider "Business" or leave it as Admin. Your call to confirm or adjust.
OQ-3 Launch Hub dormant state — confirm the design approach
Launch Hub will be empty for most of the year between active release cycles. The UX Review identifies this as "the temporal gap problem" and recommends an explicit dormant state design: a callout at the top of Launch Hub reading "No active release. This hub activates when a release is 60 days out." with a single button to create a new Release record. Phase 3 (Dashboard Design) will design the Launch Hub layout — it needs this decision locked beforehand so the dormant state is designed intentionally rather than as an afterthought.
Recommendation Confirm the dormant mode design: top of Launch Hub shows a single callout block with the activation message and one "New Release →" button. Everything else is hidden or shows empty state. This approach is honest, clean, and prevents the "abandoned hub" feeling. Confirm to lock for Phase 3.
OQ-4 The Book of JaZeR — rebuild or keep as-is inside Song Lab?
The Book of JaZeR exists as an active page in the current workspace and is confirmed to live inside Song Lab. The question is whether it stays exactly as it is (moved into Song Lab, no changes to its internal structure) or whether it gets rebuilt as a cleaner template-driven sub-page during the Phase 7 build. This matters because Phase 4 (Templates) must know whether to produce a Book of JaZeR template or treat it as a pre-existing page with no Phase 4 deliverable.
Recommendation Keep as-is for now. Move it into Song Lab during Phase 7 without restructuring. Evaluate its internal structure after 14 days of use alongside the new Songs DB and Writing Assets DB — if it needs reworking, do it then with real usage data. Confirm to exclude it from Phase 4 template work.
