01 — Database Model Verdict

Structural Score

DB Score
6.5
out of 10
Schema Incomplete
Assessment

The five-database model is correct. The decision not to split Writing Assets into five separate databases is architecturally sound and will be defended below. The relational spine (Songs as the hub, all other databases orbiting it) is the right topology.

The score drops because the formula layer is entirely absent, the rollup selection is undefined, the property naming is inconsistent across databases, the Sessions → Writing Assets relation should be cut, and no templates are specified — meaning the system will be built, filled with data, and then discovered to have structural problems that require teardown.

Biggest database risk: The Tasks DB without a Focus property and without an Overdue formula will generate a growing backlog of mixed-type tasks that no filtered view can surface cleanly. The result is a database people stop checking — which means tasks pile up, nothing gets done, and the whole task layer gets abandoned. This failure happens at the view architecture level, not the schema level. Fix it at the property level before the first record is created.

After the corrections in this document, the model should reach 8.5/10. The remaining gap reflects that real-world Notion limitations (formula date arithmetic, rollup on filtered relations) will require minor adjustments during build.

05 — Formula Recommendations

Automated Intelligence Layer

These formulas transform static databases into active decision surfaces. All use Notion's formula syntax. Build them after the rollup properties they depend on are confirmed working.

Days Stale
Songs DB → requires: Last Session (rollup)
Returns the number of days since the last logged session. The raw integer behind the Stale Warning — also useful as a sortable property in the "Stale Songs" view.
if(prop("Last Session") != empty, dateBetween(now(), prop("Last Session"), "days"), 999 // 999 = never worked on; surfaces at top of stale sort )
Surfaces in: Stale Songs view (sort descending) · Weekly Review · Tooltip on song cards
Stale Warning
Songs DB → requires: Days Stale (formula) · Status
Flags songs that are active in the pipeline (not Concept, not Released, not Shelved) but haven't had a logged session in 14+ days. This is the accountability mechanism that prevents songs from silently dying.
if( and( prop("Status") != "Concept", prop("Status") != "Released", prop("Status") != "Shelved", prop("Days Stale") > 14 ), "⚠️ Stale", "" )
Surfaces in: Active Songs board (visible property) · Stale Songs filtered view · Weekly Review checklist
Release Health
Releases DB → requires: Days Until Release (formula) · Open Release Tasks (rollup) · Artwork Done · Promo Assets Ready
Produces a three-tier health signal for releases in progress. Answers "is this release on track?" without manually checking four properties.
if( and( prop("Days Until Release") < 21, or( prop("Open Release Tasks") > 3, prop("Artwork Done") == false, prop("Promo Assets Ready") == false ) ), "🚨 At Risk", if( prop("Days Until Release") < 60, "⚡ Active", "📅 Planned" ) )
Surfaces in: Release Pipeline view (visible as first property after Status) · Master Dashboard release widget
Days Until Release
Releases DB → requires: Release Date
Simple integer countdown. Negative values indicate overdue releases. Used by Release Health and as a sort key in the Pipeline view.
if( prop("Release Date") != empty, dateBetween(prop("Release Date"), now(), "days"), 9999 // No date = treat as far future; sorts to bottom )
Overdue
Tasks DB → requires: Due Date · Status
Flags tasks with a past Due Date that aren't marked Done. Surfaces in the Due This Week view as a red marker. The most important signal in the Tasks DB — overdue tasks need immediate resolution or deletion.
if( and( prop("Due Date") != empty, dateBetween(prop("Due Date"), now(), "days") < 0, prop("Status") != "Done" ), "⚠️ Overdue", "" )
Surfaces in: Due This Week view · Focus Board (sorted to top) · Weekly Review task purge
Auto Session Name
Sessions DB → requires: Session Type · Date
Eliminates the naming step entirely when creating a session. Reduces logging friction to two required inputs (Type and Date). The name is generated automatically — no typing required.
if( prop("Session Type") != empty, prop("Session Type") + " — " + formatDate(prop("Date"), "MMM D, YYYY"), "New Session" ) // Result: "Writing — May 16, 2026"
⚠️

Notion requires the Title property to be a Title type, not a Formula type. Work around this: keep Title as a regular Title property but make it optional in the template, and add the Auto Name formula as a separate Formula property used for display in views. Use the Formula property as the display name in gallery and board views.

Floating
Writing Assets DB → requires: Song (relation) · Status
Boolean that powers the Unattached Ideas view without requiring a complex filter setup on every view. If Floating = true, the asset has no Song and isn't Archived — it needs attention.
and( empty(prop("Song")), prop("Status") != "Archived" )
Surfaces in: Unattached Ideas view (filter: Floating = true) · Dashboard Quick Capture widget
06 — Database Simplification

Remove, Merge, Delay

ItemActionReason
Sessions → Writing Assets relation REMOVE Maintenance overhead with zero workflow return. The session page body handles "what was created." The Writing Asset's Song relation handles lineage. This relation solves a problem that doesn't exist in practice.
"Type" property naming across all databases RENAME Three databases currently use "Type" — Writing Assets, Sessions, Tasks. When building filters or formulas, "Type" is ambiguous in the context of a multi-database view. Rename to "Asset Type," "Session Type," "Task Type" — unambiguous in any context.
Song Status "Idea" RENAME → "Concept" Naming collision with Writing Assets Type = "Idea." A Song at this status is a named, intentional project placeholder. A Writing Asset of type Idea is a fragment. Different objects need different names.
"Mood/Vibe" or "Mood/Energy" properties CONSOLIDATE → "Energy" Both "Mood" and "Vibe" are subjective and hard to filter consistently. "Energy" (High/Medium/Low/Variable) is measurable, actionable, and consistent across Songs and Writing Assets. Use one property name, one option set, everywhere.
Sessions DB Status property DO NOT BUILD Sessions are event records. They don't have a "Planned/Done" status — that's task thinking applied to event logging. If you want to plan future sessions, create a Task (Type = Production) instead. Adding a Status to Sessions creates a to-do system inside an event log.
Writing Assets "Concept" type option RENAME → "Concept Note" Same naming collision issue. A Concept Note (Writing Asset) is a description of what a song is about. A Concept (Song Status) is a song project placeholder. Renaming to "Concept Note" disambiguates.

Finance / Income tracking DELAY indefinitely No database until streaming income or sync licensing requires structured tracking. A simple linked spreadsheet is sufficient until the bottleneck is tracking, not volume.
Collaborators DB DELAY until active Build when there are 5+ active collaborators requiring tracking. Until then, a "Collaborators" section on the Brand Reference page handles the use case with zero maintenance overhead.
07 — Build Sequence

Exact Order and Dependencies

🔒

Pre-build requirement: Before creating any database in Notion, write down the exact filter conditions for your three most-used views: Focus Board (Tasks), Unattached Ideas (Writing Assets), and Active Songs (Songs). If you can't describe those filter conditions from memory, the build isn't ready. Property decisions made during build under time pressure are the primary source of structural inconsistency.

1
Songs DB — schema only, no relations yet
Build the title property, all Status options (including Concept and Shelved), Priority, BPM, Key, Energy, Song Type, Genre Tags. Apply the Full Track template spec. Create Active Songs board view and Writing Queue view. Migrate the 4 active songs (Transformative Trance, Mission Statement, Divine Rerun, LIFE). Do not build any relations until Step 3.
Dependencies: None · Duration: 1–2 hours
2
Writing Assets DB — schema only, no relations yet
Build Asset Type (8 options, correctly named), Status flow, Date Created (Created Time auto), Energy, Genre Fit, Recorded checkbox. Set default view = Unattached Ideas with Floating formula pre-configured. Build Hook Bank view. Create Quick Idea, Freestyle, and Lyric Sheet templates. Do not build the Song relation until Songs DB is confirmed complete.
Dependencies: None · Duration: 1–2 hours
3
Songs ↔ Writing Assets relation
Create the relation between Songs and Writing Assets. Verify the reverse relation appears correctly on Writing Assets as "Song" (singular). Test by linking a Writing Asset to one of the 4 active songs. Confirm the Unattached Ideas view correctly filters to assets with no Song linked. This is the most important relation in the system — test it thoroughly before adding more.
Dependencies: Steps 1 + 2 · Duration: 30 minutes
4
Sessions DB — minimal schema
Build Name (title), Session Type (5 options), Date. Add optional properties below the fold: Energy, Key Win, Next Step, Duration. Build the Songs relation immediately (it's the core of Sessions usefulness). Create all 4 session templates. Confirm each template defaults Date to today. Test with one real session log. Do NOT add any rollups to Songs DB yet — wait for 10+ sessions.
Dependencies: Step 1 (Songs DB must exist for the relation) · Duration: 1 hour
5
Tasks DB — full schema with Focus property
Build Task (title), Task Type (6 options), Status (4 options), Focus checkbox, Priority. Add optional: Due Date, Notes. Build Songs and Releases relations (Releases DB doesn't exist yet — add the relation property now as a placeholder; it will resolve when Releases is created in Step 7). Build Focus Board view and Due This Week view with exact filter specs locked. Create Production Task template.
Dependencies: Step 1 (Songs relation) · Duration: 1–2 hours
6
Songs ↔ Tasks relation + Overdue formula
Link Songs and Tasks databases. Test by creating one Production Task for an active song and confirming it appears in the Songs DB Tasks relation. Build the Overdue formula on Tasks. Build the Tasks linked view inside a Song page body (filtered to that Song). Confirm the filter works correctly.
Dependencies: Steps 1 + 5 · Duration: 45 minutes
7
Releases DB — build only when a release is 60 days out
Build full Releases schema: Title, Status, Format, Release Date, all optional properties, Songs relation, Tasks relation. Build Days Until Release formula and Release Health formula.
