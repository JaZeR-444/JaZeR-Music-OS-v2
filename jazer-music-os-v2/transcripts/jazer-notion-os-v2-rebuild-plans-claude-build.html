01 — Workspace Architecture

The Hub System

The workspace is organized around one master dashboard and five focused hubs. Nothing more. Each hub has exactly one job. When a resource doesn't clearly belong to a hub, it goes to the Vault.

Architecture rule: The Master Dashboard surfaces filtered views from the databases — it does not contain content itself. The hubs organize access to full databases. The databases hold the actual data. These three layers must stay distinct or the system collapses into an archive.

02 — Database Ecosystem

The 5 Core Databases

Every database below was put through a strict filter: does it improve workflow, clarity, or scalability enough to justify maintenance overhead? If not, it was cut. The result is five databases. No more.

⛔ Databases that were cut and why
  • Plugins DB → already live in FL Studio Plugin CC with 141 rows. Don't duplicate.
  • Samples/Sounds DB → reference only. Lives in FL Studio hub as a page.
  • Collaborators DB → solo artist. A contacts page is sufficient until a team emerges.
  • Finance / Income DB → no data volume yet. Use a simple spreadsheet until streaming revenue is tracked regularly.
  • Hooks DB → hooks are a Type in Writing Assets, not a separate database.
  • Ideas DB → same: a Type in Writing Assets.
  • Lyrics DB → same: a Type in Writing Assets.

03 — Creative Database Strategy

The Unified Writing Assets Decision

This is the most important architecture choice in the system. Here are the three realistic options and the tradeoffs:

OptionStructureStrengthWeakness
A — Separate DBs Ideas DB + Lyrics DB + Hooks DB + Freestyles DB Clean visual separation per content type An idea becomes a hook becomes a verse. 4 databases means you track the same asset in 3 places as it evolves. Maintenance nightmare.
B — Unified DB One Writing Assets DB with a Type property An asset evolves in one place. Type changes. Relations stay intact. Zero double-entry. Requires discipline with the Type property. The database gets large, so you need strong views to filter it.
C — Hybrid Songs DB + separate Freestyles DB, everything else unified Freestyles feel distinct enough that separation has some logic This is a compromise that brings back the weakness of Option A for a very narrow gain. The Freestyle Session template already distinguishes them as a Type, which is sufficient.
Final Decision — Option B: One Unified Writing Assets DB

A freestyle that turns into a verse that becomes the hook of a song should be one record that changes its Type and gets linked to a Song — not three separate records in three databases. The Type property handles what 4 databases were doing. The views handle the filtering.

Your existing Freestyle Hub structure is preserved as a filtered view (Type = Freestyle) inside Writing Assets, not as a separate database. The Book of JaZeR becomes the home page of Song Lab, surfacing Writing Assets linked to active songs.

Why Sessions is Separate

Sessions are time-based events. A Session record answers: "What happened on this date? How long? What was produced?" Writing Assets answer: "What exists creatively?" These are fundamentally different objects. Merging them would mean you couldn't answer "how many writing sessions did I have in April" without parsing non-event records. Sessions stay separate.

04 — Creative Workflow

Idea → Release

This is the full lifecycle map. Every stage identifies what happens, where it lives, how it's tracked, and what you do next.

Capture Writing Assets DB
Voice memo, lyric fragment, title idea, groove concept. Create a Writing Asset record (Type = Idea or Hook). Two sentences max. The goal is speed — never let capture become editing.
→ Status: Raw. No song attached yet. Surfaces in "Unattached Ideas" dashboard view.
Develop Writing Assets + Sessions
Flesh out the idea over one or more writing sessions. Create a Session record (Type = Writing). Attach Writing Assets created during the session. Update asset Status to Developing.
→ Status on asset: Developing. Appears in "Writing Queue" dashboard view.
Write Songs + Writing Assets
Once an idea has enough to become a song project: create a Song record (Status = Writing). Link Writing Assets to the Song. From here, all lyrics, verses, and sheets are created as Writing Assets attached to this Song record.
→ Song Status: Writing. Appears in "Active Songs" view. Writing Assets status: Attached to Song.
Record Songs + Sessions
Log a Session (Type = Recording). Link it to the Song. Notes on takes, mic chain, energy level. The Session record is the paper trail. The Song record's status advances.
→ Song Status: Recording. Appears in Production Hub's "Production Queue" view.
Produce Songs + Tasks + Sessions
Production work happens in FL Studio. Each major production task (drum programming, bass, arrangement, sound design) gets a Task record (Type = Production) linked to the Song. Reference plugins from the FL Studio Plugin CC via link — do not re-enter plugin data.
→ Song Status: Producing. Tasks surface in Production Hub's task view.
Review Songs + Sessions
Critical listening session. Log as a Session (Type = Mix Review). Key outputs: notes on what needs fixing, LUFS target, reference track comparison. Update Song record with LUFS reading. Use Critical Coaches framework from existing resource.
→ Song Status: Mixing → Review (cycle as needed).
Release Prep Releases + Tasks
Create a Release record. Link the Song. Generate release prep Tasks (Type = Release Prep): artwork, metadata, distribution submission, pitch deadline. Song status advances to Release Prep.
→ Release Status: Prep. Tasks surface in Launch Hub's release task view.
Publish Releases + Songs
Distribution submitted. Release date locked. Mark Release Status = Submitted → Live. Update Song Status = Released. Add distribution link and ISRC to Song record.
→ Song moves to Catalog view. Release status: Live.
Promote Tasks + Releases
Content Tasks linked to the Release: IG posts, short-form video, playlist pitching, press outreach. Generated from the Content Calendar Generator tool. Track as Tasks (Type = Content).
→ Release Status: Promoting. Tasks surface in Launch Hub content view.
Analyze Songs + Releases
After 30/60/90 days: add a review note to the Song record. Stream count snapshot (manual or via linked Drive doc). What worked? What to do differently? This feeds the next cycle.
→ Song stays in Released Catalog. Release archived. Insights loop back into next writing session.
05 — Production / FL Studio

Linked Hub, Not Embedded Archive

Architecture Decision

The FL Studio Plugin Command Center stays exactly where it is — as a sibling hub with its own 7 databases (141 plugins, parameters, presets, plugin families, scorecard, build tasks, document index). It does not move inline into Music OS v2. It does not get simplified. It does not get archived.

What changes is how Music OS v2 connects to it: lightweight link in the sidebar, a single linked view of production tasks in the Production Hub, and a reference link in each Song record's page body. That's the entire integration surface.

What Stays in FL Studio Plugin CC

All 7 inline databases Plugin scorecard system Build task history Parameter documentation Plugin families Presets reference Document index Producer's Guide trilogy (Drive) Sample pack licenses

What Appears in Production Hub (Music OS v2)

WhatHowWhy
Songs in production Filtered view of Songs DB (Status = Recording / Producing / Mixing) See what's in the pipeline without going to FL Studio hub
Production Tasks Filtered Tasks view (Type = Production) Actionable production checklist without plugin reference clutter
FL Studio Plugin CC link Linked mention / button at top of Production Hub One click to the full plugin system when needed
Plugin reference (per Song) Manual notes block inside each Song page body Quick "which chain did I use" note without importing the DB
🎛️

The rule: If it's about a plugin as a tool, it lives in FL Studio Plugin CC. If it's about a specific song's production, it lives in a Task or a note inside the Song record. The two systems should never blur.

07 — Relational Database Map

How Everything Connects

Songs DB
↔ Writing Assets DB ← Many-to-many. A song can have many assets; an asset can float before being attached.
↔ Sessions DB ← Many-to-many. A song has many sessions; a session can cover multiple songs.
→ Tasks DB ← One-to-many. One song spawns many production and admin tasks.
→ Releases DB ← Many-to-one (usually). Multiple songs can appear on one EP/album release.

Releases DB
↔ Songs DB ← Synced relation (reverse of above)
→ Tasks DB ← One-to-many. Release prep + promo tasks link to a Release.

Sessions DB
↔ Songs DB ← Synced relation
→ Writing Assets DB ← Optional: link assets created in a session to that session record.

Writing Assets DB
→ Songs DB ← Synced relation (reverse of Songs → Writing Assets)

Tasks DB
→ Songs DB ← Optional relation (production tasks)
→ Releases DB ← Optional relation (release prep / promo tasks)

— — — — EXTERNAL CONNECTIONS — — — —

FL Studio Plugin CC
· Not relationally linked. Connected by URL reference inside Song page bodies and Production Hub sidebar link. Intentionally NOT a Notion relation — the plugin system is a standalone reference hub, not part of the data model.

Google Drive (Brand / Persona / Guides)
· Referenced by URL embed in Brand Hub and Release pages. Not a Notion relation. Drive holds large PDFs and reports; Notion holds structured metadata and tasks.
